# Story 2.1 — Segurança: service_role_key + Carregamento Pré-Auth

**Epic:** 2.0 — Hardening & Production-Ready
**Status:** Done
**Complexidade:** S (Small)
**Agente:** @dev

## Objetivo
Remover a service_role_key exposta ao browser e corrigir o carregamento de dados que ocorre antes da autenticação, eliminando erros 403 no console.

## Acceptance Criteria

### Service Role Key
- [x] AC1: `VITE_SUPABASE_SERVICE_ROLE_KEY` removido do `.env`
- [x] AC2: Nenhuma referência a `SERVICE_ROLE_KEY` no código-fonte client-side

### Carregamento Condicional
- [x] AC3: `useAppState` aceita parâmetro `enabled` (boolean) — só carrega quando `true`
- [x] AC4: `App.tsx` passa `enabled` baseado no estado de autenticação (user !== null)
- [x] AC5: Zero erros 403 no console durante fluxo de login

### Validação
- [x] AC6: Build sem erros (`npm run build`)
- [x] AC7: Testes passando (`npm run test`)

## Arquivos Impactados
- `.env` — remover SERVICE_ROLE_KEY
- `hooks/useAppState.ts` — adicionar parâmetro `enabled`
- `App.tsx` — passar `enabled` baseado em auth state

## Notas Técnicas
- O `VITE_SUPABASE_SERVICE_ROLE_KEY` não é usado em nenhum arquivo, mas sua presença no `.env` com prefixo `VITE_` o expõe ao browser via `import.meta.env`
- O `useAppState()` atualmente carrega dados via `loadState()` no `useEffect` imediatamente ao montar, antes do `onAuthStateChange` resolver o estado de auth
- A correção deve garantir que `loadState()` só execute quando houver um usuário autenticado
